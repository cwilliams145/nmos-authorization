NMOS Authorization API documentation

https://api.example.com/{issuer_id_path}

  • issuer_id_path: required(string)

    The OPTIONAL path component of the issuer ID, in accordance with Section 3 of RFC 8414.

Overview

The NMOS Authorization API is exposed by an Authorization Server as a standard interface for requesting, revoking, and managing tokens and client registration.

This is an example implementation only. Authorization Servers may use alternative names for API paths provided these are reflected in the Authorization Server Metadata.

Endpoint Discovery

The Authorization Server URL is discoverable using unicast and/or multicast DNS using the '_nmos-auth._tcp' service name.

Registering Clients

The /register endpoint should be protected by some appropriate authentication scheme to limit the availability of client-registration to valid clients. This is implementation-specific. This may be via the use of user credentials, public/private keys or an initial JWT token. This equally may be administered by an enterprise system such as a Single-Sign-On system. Please see RFC7591.

Use of HTTPS/TLS

The Authorization Server MUST require the use of TLS when interacting with the /token and /authorize endpoints.
